## Часть 1: Создание протокола в Confluence

### Шаг 1: Спросить куда сохранить

При КАЖДОМ вызове спрашивай:

```text
Куда сохранить протокол в Confluence?
Варианты:
1. ID родительской страницы
2. URL страницы (скопируй из браузера)
```

### Шаг 2: Спросить тип протокола

Перед анализом транскрипции ОБЯЗАТЕЛЬНО спроси пользователя, какой тип протокола нужен:

```text
Какой тип протокола нужно подготовить?
1. Стандартный протокол
2. Расширенный протокол
```

Если доступен `question` tool, используй его с двумя вариантами выбора.

**Правила:**
- Если выбран **Стандартный протокол** — извлекай только базовые сущности и не задавай дополнительные уточняющие вопросы, кроме критически необходимых
- Если выбран **Расширенный протокол** — обязательно задай дополнительные уточняющие вопросы, если в транскрипции не хватает данных для качественного управленческого протокола
- Сохрани выбор в переменную `protocol_type` со значением `standard` или `extended`

### Шаг 3: Прочитать транскрипцию

Извлеки из файла базовые данные:

- Дату встречи
- Название встречи
- Участников (список ФИО)
- Ключевые темы
- Принятые решения
- Задачи с ответственными и дедлайнами

#### Если выбран "Стандартный протокол"

Сформируй краткий и структурированный протокол без лишних уточнений.

Включи:
- Дату
- Название встречи
- Участников
- Цель встречи (кратко)
- Повестку
- Принятые решения
- Задачи с ответственными и дедлайнами
- Краткое резюме

Задавай уточняющие вопросы только если без них нельзя корректно сформировать протокол:
- неясна дата
- непонятно, кто ответственный
- задача совсем неконкретна
- невозможно определить, было ли принято решение

#### Если выбран "Расширенный протокол"

Сформируй управленческий протокол повышенной детализации.

Обязательно дополнительно извлеки или уточни:

1. **Контекст / Предыстория**
   - откуда взялась встреча
   - что было до (предыдущие встречи, решения, документы)
   - причина созыва

2. **Кворум**
   - кто присутствовал (фактически)
   - кто должен был быть (по плану)
   - кто отсутствовал и по какой причине (если известно)

3. **Детальную цель встречи**
- бизнес-контекст
- проблема или возможность
- ожидаемый результат встречи
- какой уровень решения ожидался: информирование / выработка предложений / принятие решения / эскалация

2. **Статус обсуждения по вопросам**
- решено
- требует доработки
- отложено
- вынесено на эскалацию

3. **Подробные action points**
Для каждой задачи зафиксируй:
- что именно нужно сделать
- какой ожидается результат или артефакт
- кто персонально отвечает
- кто соисполнители, если есть
- срок
- критерий готовности

4. **Спорные и открытые вопросы**
Для каждого такого пункта выдели:
- суть спорного вопроса
- позиции сторон
- что мешает договориться
- следующий шаг
- нужна ли эскалация

5. **Риски и зависимости**
Если они явно звучали во встрече, извлеки:
- риск
- влияние
- зависимость от другой команды, решения или срока
- что нужно сделать для снятия риска

6. **Ожидаемые follow-up действия**
- что нужно проверить после встречи
- какие материалы должны быть досланы
- какие следующие встречи или контрольные точки нужны

#### Правила формулировки цели встречи для расширенного протокола

Цель встречи должна быть максимально конкретной и не должна ограничиваться фразами вроде:
- "обсудить статус"
- "синхронизироваться"
- "рассмотреть вопрос"

Формулируй цель в структуре:
- бизнес-контекст
- предмет обсуждения
- ожидаемый итог встречи
- горизонт решения

Пример хорошей формулировки:

```text
Цель встречи: согласовать целевой вариант запуска MVP витрины X, определить зоны ответственности между командами Less, аналитики и бизнеса, а также зафиксировать перечень действий и сроков для выхода на подготовку релиза до 01.09.2026.
```

Если цель встречи в транскрипции размыта, задай уточняющие вопросы:

```text
Для расширенного протокола нужно уточнить цель встречи:
1. Какую основную бизнес-проблему или задачу вы хотели решить?
2. Какой результат считался успешным по итогам встречи?
3. Вы хотели принять решение, подготовить предложения, синхронизировать позиции или эскалировать вопрос?
```

#### Правила формулировки action points

Каждая задача должна быть сформулирована так, чтобы из неё было понятно:
- кто делает
- что делает
- какой должен быть результат
- к какому сроку
- как понять, что задача выполнена

Плохо:
- "разобраться"
- "обсудить"
- "подготовить вариант"

Хорошо:
- "Иванов И.И. готовит и направляет участникам встречи сводный список интеграционных ограничений по системе X до 25.07.2026; результатом считается документ в Confluence с подтверждением от архитектуры."

Если задача в транскрипции неконкретна, задай уточняющий вопрос по недостающим полям.

#### Правила уточняющих вопросов для расширенного протокола

Если данных недостаточно, ОБЯЗАТЕЛЬНО задай уточняющие вопросы пользователю.

Задавай вопросы только по недостающим данным, а не всем списком сразу.

Приоритет уточнений:
1. цель встречи
2. спорные вопросы
3. action points
4. сроки и ответственные
5. риски и зависимости

Пример:

```text
Для расширенного протокола не хватает нескольких уточнений:

1. Цель встречи:
- какую основную проблему вы хотели решить?
- какой результат считался успешным по итогам встречи?

2. По задаче "подготовить вариант":
- кто основной ответственный?
- какой должен быть результат: документ, согласование, расчёт, презентация?
- какой срок?

3. По спорному вопросу о приоритетах:
- в чём различались позиции сторон?
- нужен ли вынос на эскалацию?
```

### Шаг 4: Сформировать HTML протокола

Используй шаблон в зависимости от значения `protocol_type`:
- `standard` -> стандартный HTML-шаблон
- `extended` -> расширенный HTML-шаблон

Перед генерацией HTML проверь:
- заголовок страницы содержит дату и краткое название встречи
- цель встречи заполнена осмысленно
- в задачах нет безличных формулировок
- у каждой задачи есть ответственный
- спорные вопросы не смешаны с принятыми решениями

### Шаг 5: Создать страницу в Confluence

Используй Python + requests:

```bash
uv run --with requests python -c "
import requests, json
token = 'TOKEN'
url = 'https://confluence.moscow.alfaintra.net/rest/api/content/'
page_data = {
    'type': 'page',
    'title': 'ДД.ММ.ГГГГ - Название',
    'ancestors': [{'id': 'PARENT_ID'}],
    'space': {'key': 'ADIR'},
    'body': {'storage': {'value': 'HTML', 'representation': 'storage'}}
}
r = requests.post(url, headers={'Authorization': f'Bearer {token}', 'Content-Type': 'application/json'}, data=json.dumps(page_data), verify=False)
print(r.json().get('id'))
"
```

## Часть 2: Отправка писем через Outlook

### Перед отправкой — показать редактируемую таблицу

Сформируй редактируемую таблицу с полями:

```text
| Почта | ФИО | Задача | Дедлайн |
|-------|-----|--------|---------|
| email@investlab.tech | Фамилия И.О. | Описание задачи | ДД.ММ.ГГГГ |
```

**СПОСОБ РЕДАКТИРОВАНИЯ:**
Покажи данные в формате, где пользователь может:
1. **Указать корректную почту**, если найденная неверна
2. **Отредактировать текст задачи**, если нужно уточнить
3. **Добавить или изменить дедлайн**
4. **Удалить строку**, если отправка не нужна

Примеры форматов для разных ситуаций:

**Простой вариант (текстом):**

```text
Отредактируй перед отправкой:
1. ivanov@investlab.tech | Иванов И.И. | Подготовить отчёт | 25.06.2026
2. petrov@alfabank.ru | Петров П.П. | Проверить данные | -
---
(напиши "ок" или укажи изменения)
```

**Рекомендуемый вариант — используй question tool с custom: true:**
Позволяет пользователю ввести свой вариант с исправлениями прямо в текстовом поле.

### Правило перед рассылкой писем

Перед формированием писем убедись, что каждая задача содержит минимум:
- ответственного
- понятную формулировку действия
- срок или отметку, что срок не определён
- контекст поручения

Если выбран **Расширенный протокол**, дополнительно включай в письмо:
- ожидаемый результат или артефакт
- критерий готовности
- ссылку на протокол

### Получение почты

Outlook использует домены `@investlab.tech` или `@alfabank.ru`. Поиск почты:

**Если в транскрипции указаны ФИО:**
1. Попробуй найти в AD по имени
2. Спроси пользователя: `Не нашёл почту для [ФИО]. Укажи email:`

**Если почта не найдена** — спроси у пользователя.

### Поиск email в Outlook

**Используй Python с pypiwin32:**

```bash
uv run --with pypiwin32 python -c "
import win32com.client
import pythoncom

pythoncom.CoInitialize()
outlook = win32com.client.Dispatch('Outlook.Application')
ns = outlook.GetNamespace('MAPI')
al = ns.AddressLists.Item('All Users')

users = []
for e in al.AddressEntries:
    try:
        u = e.GetExchangeUser()
        if u:
            users.append({'name': u.Name, 'email': u.PrimarySmtpAddress})
    except:
        pass

found = [u for u in users if 'ФАМИЛИЯ' in u['name']]
for u in found:
    print(u['name'], '=>', u['email'])
"
```

### Отправка письма

```bash
uv run --with pypiwin32 python -c "
import win32com.client
import pythoncom

pythoncom.CoInitialize()
outlook = win32com.client.Dispatch('Outlook.Application')
mail = outlook.CreateItem(0)

mail.To = 'email@domain'
mail.Subject = '[Протокол] Название встречи - ДД.ММ.ГГГГ'

body = '''Добрый день, [ФИО]!

По итогам встречи \"[Название встречи]\" от ДД.ММ.ГГГГ вам поручено:

[ЗАДАЧА]

[ДЕДЛАЙН]

[ДОПОЛНИТЕЛЬНЫЙ КОНТЕКСТ]

Протокол: [URL]
'''
mail.Body = body
mail.Send()
print('OK')
"
```

### Инструкции по отправке

1. **Собери список задач** из столбца `Решения и action points` или `Решения и задачи` протокола
2. **Для каждого ответственного:**
   - Определи его email, спроси если не найден
   - Сформулируй текст поручения
   - Для расширенного протокола включи результат, срок и критерий готовности
3. **Покажи редактируемую таблицу** и спроси через `question` tool с `custom: true`
4. **Жди подтверждения и исправлений пользователя**
5. **Только после подтверждения** отправь письма

### Пример использования question tool для редактирования

```python
question({
  questions: [{
    question: "Проверь и исправь получателей перед отправкой:",
    header: "Получатели писем",
    options: [
      {
        label: "Отправлять не нужно",
        description: "Не отправлять письма"
      },
      {
        label: "Отправить всем ( ok )",
        description: "Отправить письма указанным получателям"
      }
    ],
    custom: true
  }]
})
```

**Пользователь может:**
- Нажать `Отправлять не нужно`, если отправка не нужна
- Нажать `Отправить всем`, если всё верно
- Ввести свой вариант с исправлениями
