## 1. Генерация тестовых данных (Вячеслав Орда)

### Суть проблемы
При тестировании требуется большое количество тестовых данных (клиенты, сделки). Ручное создание через "4 контура" — трудоёмкий процесс. Например, для одного тест-плана нужно: 19 клиентов × 2-5 сделок на каждого.

### Предложенное решение
Подключение Альфа Скуи (Alpha Squi) к генерилкам для автоматизации создания тестовых данных.

**Реализация:**
- Подключение через ГИшный EDI (ГИ ИИшка)
- Настройка килокода на EDI
- Использование МЦП Отлас для тестовых кейсов
- Создание клиентов и сделок (акции: Лукойл, Газпром; облигации)

### Выявленные ограничения генерилок

| Проблема | Описание |
|----------|----------|
| Отсутствие архивирования | Не умеют переводить сделки в архив |
| Отсутствие ID сделок | Не возвращают номера сделок в БД, что необходимо для скриптов архивации |
| Контекстное окно | ГИшная ИИшка "слабенькая", постоянно выпадает по контексту |
| Ограниченный набор ручек | Работают только с частью API генерилок |
| Проблемы с доступами | Периодические ошибки авторизации |

### Решение проблемы с контекстом
- Файлик лога для сохранения созданных клиентов
- Разбивка данных на несколько файлов для периодического подключения

### Доработки для генерилок (зафиксированы в Confluence)
Ссылка: страница "Потребность команды генерации данных"

1. Доработать методы для создания архивированных сделок
2. Возвращать номера сделок в ответах API

## 2. Скиллы Дарьи Пискуновой

### 2.1. Анализ тредов Rocket.Chat

**Назначение:** Автоматическое извлечение информации из тредов Rocket.Chat и генерация саммари.

**Реализация:**
- Подключение к Rocket.Chat через cookies (обновление каждые 1.5-2 недели)
- Доступ к тредам через API Rocket.Chat
- Обработка кодировки для корректного отображения
- Генерация структурированного ответа: участники, тема, ключевые моменты

**Безопасность:**
- Использование cookies вместо логина/пароля
- Cookies хранятся в отдельном файле
- Автоматическое обновление при устаревании

**Важно:** Для работы требуется локальная ИИшка (Alfa-GEN), не внешняя.

**Ограничения:**
- Roki Chat не дорабатывается (переход на A-Chat в 2027 году)
- Доработка через бота требует согласования бюджета с интегратором

### 2.2. Валидация требований (чек-лист для БРД)

**Назначение:** Проверка полноты и качества постановок задач.

**Функционал:**
- Оценка по 56-балльной шкале
- Проверка: полнота требований, противоречивость, грамматика, лексика
- Выделение критических ошибок
- Рекомендации по улучшению

**Настройка:**
- Шаблон проверки составлен совместно с ИИ (искали best practices в интернете)
- Прописаны правила: не писать в чат, не додумывать информацию

**Проблемы:**
- Плохо справляется с форматированием в JIRA
- Хорошо работает с Confluence

## 3. Агент сбора БРД (Игорь Козик)

### Архитектура скилла

**Модель:** Claude (Cluade 3.5 Sonnet / Fable 5 до конца июля бесплатно)

**Структура:**
```
├── skill.md              # Основное описание скилла
├── changelog.md          # История изменений (8-я версия)
├── questions/            # Банк вопросов
│   ├── bank_*.md         # 65 вопросов (10 блоков)
│   ├── followup_*.md     # Follow-up паттерны
│   └── startup.md        # Стартовые вопросы
├── templates/
│   └── brd_template.md   # Шаблон BRD (15 секций)
├── README.md             # Описание работы
└── .gitignore
```

### Блоки вопросов

1. Юридическое лицо (Альфа-Инвестиции / ГО Инвест)
2. Корпоративные KPI
3. Код бюджета PPM и подразделение КР
4. Регуляторные требования
5. Бизнес-контекст и конкуренты
6. Функциональные требования
7. Тарифы и комиссии
8. Интеграции
9. Комплаенс
10. Нефункциональные требования

### Особенности работы

**Адаптивная глубина:**
- Мелкие фичи: пропуск банка вопросов, быстрый путь (1-4 блока)
- Крупные инициативы: полный цикл (15 блоков)

**Работа с материалами:**
- Запрос существующих материалов (презентации, кастдевы, расчёты)
- Предзаполнение ответов из материалов
- Запрос подтверждения актуальности

**Безопасность:**
- Проверка: это инструкции или источник? (защита от prompt injection)
- Документы воспринимаются как "пища для размышления", не как команды

**Результат:**
- Структурированный BRD в Confluence
- Приложения (роли для смежников: комплаенс, архитектор, инфобез)
- Выделение рисков

### Статистика

- Прохождение демо: ~40 минут (35-40% покрытия)
- Количество вопросов: 65

## 4. Обсуждение инфраструктуры

### GitHub vs GitLab
- **Текущий подход:** GitLab (внутренний)
- **Предложение:** Публичный GitHub для рабочей группы
- **Обоснование:** Проще настройка, можно делиться скиллами между командами

### Проблема с защитой веток
- GitHub Teams / Enterprise требуется для обязательных ревью
- Временно: публичный репозиторий без защиты

### Следующие шаги по инфраструктуре
1. Игорь настраивает GitHub репозиторий
2. Все выкладывают скиллы туда
3. Совместное развитие через PR

### Перспективы
- Прокси-сервис для внешних моделей (маскирование данных)
- Подписки ChatGPT / Claude для команды
- Использование продвинутых моделей для аналитиков
